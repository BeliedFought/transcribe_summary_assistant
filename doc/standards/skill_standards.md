# Стандарт оформления навыков AI-агента. Версия 4.9.0. 2026-07-21

Документ описывает правила создания и оформления навыков (skills) - специализированных инструкций для AI-агента. Является дополнением к `doc/standards/project_standards.md` и применяется поверх него. Все требования основного стандарта остаются в силе, кроме случаев, явно описанных в данном документе.

Навыки не заменяют `AGENTS.md`, `CLAUDE.md` и `doc/standards/project_standards.md`. В навыках - только узкие сценарии и пошаговые инструкции для конкретных задач.

**Применимость и обязанность соответствия.** Правила настоящего стандарта едины для всех репозиториев - хаба `project_standards` и производных проектов. Навык, создаваемый или редактируемый вне хаба (в производном репозитории), обязан соответствовать этому стандарту в той же полноте: обязательный frontmatter (`id`, `description`, `auto_apply`), структура разделов, именование файла, стиль и объем. Локальных исключений нет - производный репозиторий применяет стандарт как есть, без ослабления требований. Перед созданием или изменением файла в `doc/skills/` агент сверяется с настоящим документом.

Общие и хаб-навыки (`hub_sync_standards`, `hub_sync_skills`, `hub_sync_indexes`, `_index_skills_hub`, `glob_audit_repository`, `glob_update_kilo_rules`, `glob_update_indexes`) в производных репозиториях доступны только для чтения: их поддерживают в хабе и распространяют навыком `hub_sync_skills`, поэтому локальные правки будут затерты при синхронизации. Собственные проектные навыки репозитория создаются и редактируются локально с соблюдением требований настоящего стандарта.

## 1. Назначение навыка

Навык - файл с инструкцией для AI-агента по одной специализированной задаче или workflow.

Примеры: синхронизация стандартов между проектами, развертывание по внутреннему чеклисту, генерация отчета в фиксированном формате.

Навык создавать только если сценарий:
- повторяется в проекте;
- требует доменных знаний, которых нет в общих правилах;
- описывается пошагово, с командами, путями или шаблонами.

## 2. Расположение файлов

*Обязательно.*

Каталог навыков - `doc/skills/`. Он имеет двухуровневую структуру:

- **Корень `doc/skills/`** - общие навыки, применимые к любому проекту (пример: синхронизация стандартов, аудит репозитория). Эти навыки входят в универсальный набор и работают в любом репозитории.
- **`doc/skills/<repo-name>/`** - проект-специфичные навыки, применимые только к конкретному проекту. `<repo-name>` - имя папки репозитория (`Path.cwd().name`), одна подпапка на проект.

```
doc/skills/
├── glob_audit_repository.md   # общий навык (префикс glob_, раздается всем)
├── hub_sync_standards.md      # хаб-навык (только в хабе, префикс hub_)
└── my_project/                # проект-специфичные навыки
    └── db_migrate.md
```

**Правила:**

- Общие навыки размещать только в корне `doc/skills/`.
- Проект-специфичные навыки размещать только в `doc/skills/<repo-name>/`, где `<repo-name>` - имя папки текущего репозитория.
- Хаб-навыки (служебные навыки хаба `project_standards`, которые не распространяются в производные репо) размещать только в корне `doc/skills/` хаба и именовать с префиксом `hub_` (раздел 3).
- Общие (распространяемые) навыки размещать в корне `doc/skills/` и именовать с префиксом `glob_` (раздел 3); хаб раздает их во все репо навыком `hub_sync_skills`.
- Не создавать навыки в корне репозитория, в `src/` и других каталогах вне `doc/skills/`.
- Папка `doc/skills/` опциональна - создавать только если в проекте предусмотрены навыки.
- Один навык - один файл `.md`. Структура внутри уровня (корень или подпапка проекта) - плоская: без вложенных `scripts/`, без вложенных `SKILL.md`.
- Имя проектной подпапки - всегда в нижнем регистре, совпадает с именем папки репозитория.

**Почему `doc/skills/` как единая точка хранения:**

- навыки копируются в новые проекты вместе с шаблоном;
- не привязаны к конкретному инструменту или агенту;
- версионируются в git вместе с доменной логикой проекта;
- служебные каталоги инструментов остаются только для локальных настроек среды.

**Каталог-агрегатор.** Центральный репозиторий `project_standards` служит хабом каталога навыков: навык `hub_sync_skills` собирает навыки из всех репозиториев в `project_standards/doc/skills/<repo-name>/` для единого обзора. Общие навыки (корень `project_standards/doc/skills/`) поддерживаются вручную как каноничные.

**Большие навыки:** если материал не помещается в лимит объема (раздел 5), разбить сценарий на несколько файлов в `doc/skills/` (или в `doc/skills/<repo-name>/` для проект-специфичных) или вынести код и шаблоны в `run/`, `doc/specs/` - в теле навыка оставить ссылку и команду вызова.

## 3. Именование файлов

*Обязательно.*

| Правило | Пример |
| ------- | ------ |
| Latin characters, `snake_case` | `deploy_docker.md` |
| Расширение `.md` | `review_pr.md` |
| Имя отражает задачу, не общие слова | `deploy_docker.md`, не `helper.md` |
| Схема `{домен}_{действие}` или `{действие}_{объект}` | `db_migrate.md`, `parse_report.md` |
| Без пробелов, без кириллицы в имени файла | `api_test_requests.md` |

Имя файла без расширения используется как идентификатор навыка (`id`) при ссылках в чате, frontmatter и документации.

Для проект-специфичных навыков имя файла формируется так же (snake_case), а размещается файл в `doc/skills/<repo-name>/`. Имя подпапки = имя папки репозитория (раздел 2).

**Префикс `hub_` для хаб-навыков.** Навыки, которые работают только в хабе `project_standards` и не распространяются в производные репозитории (служебные навыки синхронизации и актуализации индексов), именуются с префиксом `hub_`: `hub_sync_standards.md`, `hub_sync_skills.md`, `hub_sync_indexes.md`. Префикс визуально отделяет служебные навыки хаба от общих (распространяемых) и проект-специфичных. Навык `hub_sync_skills` определяет такие навыки по префиксу `hub_` (плюс сводный индекс `_index_skills_hub.md`) и не копирует их наружу.

**Индексы каталога навыков.** Каталог имеет два индекса с разными именами: `_index_skills_hub.md` - сводный индекс хаба по всем навыкам всех репозиториев (только в хабе, ведёт `hub_sync_indexes`); `_index_skills_repo.md` - локальный индекс навыков отдельного репозитория (в каждом не-хаб репо, генерируется навыком `glob_update_indexes`). Индексные файлы (`_index_*.md`) навыками не являются и в реестры/сбор навыков не включаются.

**Префикс `glob_` для общих навыков.** Навыки, которые раздаются из хаба во все производные репозитории, именуются с префиксом `glob_`: `glob_audit_repository.md`, `glob_update_kilo_rules.md`. Навык `hub_sync_skills` выгружает наружу только файлы с префиксом `glob_` и удаляет в приёмниках устаревшие `glob_*.md`. В производных репозиториях эти файлы доступны только для чтения (перезаписываются при синхронизации). Итог: `hub_` - только хаб, `glob_` - раздается всем, без префикса в корне - не используется (проект-специфичные навыки хранятся в `doc/skills/<repo-name>/`).

## 4. Структура файла навыка

*Обязательно.*

Каждый навык - один Markdown-файл. Разделы ниже - в указанном порядке. Пустые необязательные разделы не создавать.

### 4.1. YAML frontmatter (обязательно)

Блок метаданных в начале файла - обязателен. Используется для обнаружения навыка агентом и определения режима применения. Не заменяет раздел «Когда использовать», а дополняет его. Если frontmatter отсутствует, навык считается некорректным и не применяется.

```yaml
---
id: hub_sync_standards
description: Синхронизирует doc/standards/ с соседними проектами. Применять при запросах обновить, скопировать или синхронизировать стандарты
auto_apply: true
---
```

| Поле | Обязательность | Правило |
| ---- | -------------- | ------- |
| `id` | обязательно | Совпадает с именем файла без `.md` |
| `description` | обязательно | Одна строка: что делает навык (WHAT) и когда применять (WHEN); третье лицо; ключевые слова из запросов пользователя |
| `auto_apply` | обязательно | `true` (по умолчанию) - навык применяется автоматически при совпадении запроса с разделом «Когда использовать»; `false` - только по явной ссылке `@doc/skills/` или упоминанию в `AGENTS.md`. Деструктивные навыки (удаление, перезапись, миграции БД) - всегда `false` |
| `version` | опционально | SemVer `X.Y.Z`; рекомендован для отслеживания изменений навыка |
| `status` | опционально | `draft` / `stable` / `deprecated`; дефолт `stable`. Если `deprecated` - навык не применяется |

### 4.2. Заголовок H1

Формат: `# Навык: <краткое название>`

Пример: `# Навык: Синхронизация стандартов проекта`

### 4.3. Примечание (опционально)

Блок цитаты с ограничениями применимости - только для текущего репозитория, только для определенной роли, экспериментальный статус.

### 4.4. Описание

Заголовок: `## Описание`

Один абзац: что делает навык и какой результат ожидается.

### 4.5. Когда использовать

Заголовок: `## Когда использовать`

Список из 3-5 условий или триггеров - когда агент должен применить навык. Формулировки конкретные, с ключевыми словами из реальных запросов пользователя.

### 4.6. Предусловия (опционально)

Заголовок: `## Предусловия`

Что должно быть выполнено до начала: наличие файлов, ветки, переменных окружения, прав доступа. Указывать только если без этого навык не работает.

### 4.7. Инструкция

Заголовок: `## Инструкция` или `## Инструкция по применению`

Пошаговые действия для агента. Допускаются подразделы, нумерованные списки, блоки кода, чеклисты.

Уровень детализации инструкции выбирать по хрупкости задачи:

| Свобода агента | Когда | Как писать |
| -------------- | ----- | ---------- |
| Низкая | Миграции, деплой, синхронизация данных | Точные команды и порядок шагов, без альтернатив |
| Средняя | Отчеты, PR, генерация документов | Шаблон результата и допустимые отклонения |
| Высокая | Ревью, анализ, исследование | Принципы, чеклист, критерии качества |

### 4.8. Критерии завершения

Заголовок: `## Критерии завершения`

Явный список: по каким признакам агент считает задачу выполненной. Примеры: отчет пользователю, список измененных файлов, успешный exit code команды, заполненный шаблон.

### 4.9. Примеры (рекомендуется)

Заголовок: `## Примеры`

1-2 пары «запрос пользователя → ожидаемое действие или вывод». Помогает агенту и автору навыка проверить однозначность формулировок.

### 4.10. Дополнительные разделы (опционально)

По необходимости после основных разделов:
- `## Ограничения` - что не делать;
- `## Скрипт` или вложенный код в инструкции - готовый код для выполнения;
- `## Пример результата` - образец вывода или отчета.

Встроенный код в навыке - для коротких одноразовых сценариев. Если скрипт превышает ~100 строк, вынести в `run/` или `debug/` и в навыке указать только путь и команду запуска.

## 5. Содержание и стиль

*Обязательно.*

- Язык - русский (как в основном стандарте проекта)
- Текст - лаконичный, без эмодзи, без эмоциональных формулировок
- Использовать hyphen `-` вместо em-dash и en-dash
- Не использовать ellipsis (`...`, `…`)
- Пути - через `pathlib` или явные пути относительно `PROJECT_ROOT`, как в основном стандарте
- Команды и код - в fenced-блоках с указанием языка
- Не дублировать общие правила из `project_standards.md`, `AGENTS.md`, `CLAUDE.md`
- Терминология - одна на навык; не смешивать синонимы (endpoint / route / path и т.п.)

**Объем:** навык должен быть самодостаточным для выполнения задачи, но без лишних пояснений базовых вещей. Ориентир - до 300 строк в одном файле. Если материала больше - разбить на несколько навыков по отдельным задачам или вынести код и шаблоны за пределы файла навыка (раздел 2).

## 6. Применение навыка агентом

*Обязательно.*

### 6.1. Когда применять

Агент применяет навык когда:
- пользователь явно указал файл (`@doc/skills/hub_sync_standards.md`) или название навыка;
- запрос пользователя совпадает с разделом «Когда использовать» и `auto_apply: true` в frontmatter;
- в `AGENTS.md` проекта навык перечислен как рекомендуемый для сценария.

Если frontmatter отсутствует - навык считается некорректным, не применяется. Агент сообщает пользователю о проблеме и просит добавить frontmatter.

### 6.2. Приоритет при нескольких совпадениях

1. Явная `@`-ссылка пользователя на конкретный файл навыка
2. Навык из `AGENTS.md` с более узким совпадением с запросом
3. Автоматическое совпадение по разделу «Когда использовать»
4. При равной специфичности - уточнить у пользователя; не объединять инструкции из разных навыков в один workflow

### 6.3. Порядок выполнения

Агент читает файл навыка целиком и следует инструкции. Действия навыка не должны противоречить основному стандарту и подтвержденным пользователем ограничениям проекта.

Если навык предполагает изменение файлов проекта, а пользователь не указал его явно через `@` - действуют правила уточнения (Clarification Workflow) из основного стандарта: вопросы в `doc/agent_questions/`, ожидание ответа, подтверждение перед правками.

## 7. Связь с другими документами

| Документ | Назначение |
| -------- | ---------- |
| `doc/standards/project_standards.md` | Общие правила проекта; упоминает папку `doc/skills/` |
| `doc/standards/skill_standards.md` | Правила оформления навыков (этот документ) |
| `doc/standards/_index_standards.md` | Реестр пакета стандартов; единая версия |
| `AGENTS.md` / `CLAUDE.md` | Общие инструкции агенту; реестр навыков проекта |
| `doc/skills/hub_sync_standards.md` | Синхронизация standards в соседние проекты |
| `doc/skills/hub_sync_skills.md` | Агрегация навыков из всех репо в каталог project_standards |
| `doc/skills/hub_sync_indexes.md` | Актуализация сводного индекса хаба и реестра стандартов (только в хабе) |
| `doc/skills/glob_update_indexes.md` | Актуализация локального индекса навыков репозитория (общий, во всех репо) |
| `doc/skills/_index_skills_hub.md` | Сводный индекс хаба по всем навыкам (только в хабе) |
| `doc/skills/_index_skills_repo.md` | Локальный индекс навыков репозитория (в каждом не-хаб репо) |
| `doc/skills/<repo-name>/*.md` | Проект-специфичные навыки |

В новые проекты копировать `skill_standards.md` вместе с `project_standards.md`, если в проекте предусмотрены навыки. Репозиторий `project_standards` служит центральным хабом каталога навыков: хаб-навыки (`hub_sync_standards`, `hub_sync_skills`, `hub_sync_indexes`) и общие навыки (`glob_audit_repository`, `glob_update_kilo_rules`, `glob_update_indexes`) хранятся в корне `doc/skills/` хаба; проект-специфичные агрегируются навыком `hub_sync_skills` в `doc/skills/<repo-name>/`. В производных репозиториях навыки не синхронизируются автоматически - только стандарты через `hub_sync_standards` (см. раздел 6).

**Реестр навыков в `AGENTS.md`:** каждый опубликованный навык - строка в таблице. Навык без записи в `AGENTS.md` считается черновиком и не применяется автоматически.

Навык без frontmatter не добавляется в реестр и не применяется, даже если перечислен в `AGENTS.md`.

```markdown
## Навыки проекта

| Навык | Файл | Когда применять |
| ----- | ---- | --------------- |
| Синхронизация стандартов | doc/skills/hub_sync_standards.md | обновление стандартов в соседних проектах |
```

## 8. Антипаттерны

Избегать при создании и редактировании навыков:

| Антипаттерн | Правильно |
| ----------- | --------- |
| Несколько равноправных инструментов или библиотек | Один вариант по умолчанию; альтернатива только с явным условием переключения |
| Имена `helper.md`, `utils.md`, `misc.md` | `{домен}_{действие}.md` с понятной задачей |
| Дублирование git, logging, SQL из основного стандарта | Ссылка на раздел стандарта или краткое «как в project_standards.md» |
| Time-sensitive правила («до августа 2025 используй API v1») | Актуальный метод; устаревшее - в отдельный архивный навык или раздел «Устаревшие паттерны» |
| Один навык на несколько несвязанных сценариев | Отдельный файл на каждый повторяющийся workflow |
| Длинный скрипт в теле навыка | Код в `run/` или `debug/`, в навыке - вызов |
| Размытые триггеры («когда нужна помощь») | Конкретные фразы из реальных запросов пользователя |
| Навык с `auto_apply: true` для деструктивных операций | `auto_apply: false`; деструктивные действия только по явной `@`-ссылке пользователя |

## 9. Пример

## 10. Чеклист автора

Перед публикацией навыка проверить:

- [ ] Один сценарий - один файл в `doc/skills/`
- [ ] Имя файла - `snake_case`, отражает задачу
- [ ] Frontmatter обязателен: `id`, `description`, `auto_apply` (все три поля заполнены)
- [ ] Для деструктивных навыков - `auto_apply: false`
- [ ] (Опционально) `version` и/или `status` заполнены
- [ ] «Когда использовать» - 3-5 конкретных триггеров с ключевыми словами
- [ ] «Критерии завершения» - явные признаки успеха
- [ ] Инструкция соответствует уровню свободы задачи (раздел 4.7)
- [ ] Нет дублирования общих правил из `project_standards.md`
- [ ] Объем до 300 строк или материал вынесен в `run/`, `templates/`, `doc/specs/`
- [ ] Навык добавлен в таблицу «Навыки проекта» в `AGENTS.md` и `CLAUDE.md`
- [ ] Если навык создается в производном репо - соответствие `skill_standards.md` проверено в полной мере (frontmatter, структура, именование, стиль); общие и хаб-навыки локально не редактировались

## 11. Версионирование

Версионирование пакета стандартов - единое для всех документов пакета. Политика версионирования, реестр и карта маршрутизации - в `doc/standards/_index_standards.md`. Bump версии пакета - только по явному запросу пользователя (см. _index_standards.md).
