## 01.01. Структура проекта

*Обязательно.*

Создавать только те папки, которые реально используются в проекте. Структура, правила именования и назначение папок - строгие: не создавать папки с произвольными именами, не размещать файлы не в своих папках.


| Папка  | Назначение       | Действие                                                                                                                                            |
| ------ | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| config | Конфигурация     | Файл config.ini; все настройки проекта хранить здесь. Обязательная секция [app] с name и version (раздел 01.07). Для проектов без БД - translations.json (раздел 04.01) |
| data   | Рабочие данные   | Данные для работы скриптов: SQLite, кеш, файлы состояния, справочники. Данные, которые скрипты создают и читают в процессе работы                   |
| diag   | Диагностические данные | Опциональная папка. Данные диагностики и кастомные артефакты, специфичные для конкретной машины: снимки состояния окружения, отчеты проверок, локальные отладочные дампы, результаты профилирования. Содержимое привязано к машине пользователя и не переносится между окружениями, поэтому не коммитить в git |

| doc/agent_questions | Вопросы агента   | Файлы уточняющих вопросов, создаваемые агентом в ходе согласования; шаблон questions_YYYY-MM-DD_HH-MM.md |

| sample | Примеры данных  | Примеры данных для составления спецификаций; файлы с префиксами db_, api_ и др. (префикс - ориентир по типу данных)                                |
| log    | Логи             | Создавать лог-файлы только в этой папке                                                                                                             |

**Файлы в корне:** README.md, .env, .env.example (шаблон переменных без значений), requirements.txt, .gitignore (обязателен; исключать .env, log/, debug/, diag/, input/, output/, data/, __pycache__/, .pytest_cache/, .venv/, .vscode/, .claude/, .kilo, doc/agent_questions/, doc/change_requests/). Папки `doc/standards`, `doc/specs`, `doc/skills` - коммитить. Для исключаемых папок с локальным содержимым применять паттерн `folder/*` + `!folder/.gitkeep` (см. раздел 06.01).

**Файл .env.example:** содержит все переменные из `.env` без значений. Обновлять синхронно с `.env` при добавлении или удалении переменных.

**Спецификация:** пользователь записывает требования в `doc/specs/init_spec.txt` в свободной форме; в результате общения с агентом создается `doc/specs/final_spec.md` - структурированная и согласованная спецификация.

**Папка sample:** примеры данных для составления спецификаций (init_spec.txt, final_spec.md). Файлы именуются с префиксами по типу данных: db_ - примеры структур БД/таблиц, api_ - примеры ответов API, возможны иные префиксы. Префикс служит ориентиром при выборе релевантных примеров.

**Папка run:** в run - скрипты, которые пользователь может запустить сам и получить результат. В зависимости от проекта возможны комбинации (main.py, run/, оба).

- Создавать подпапки по доменной логике: db - работа с БД, api - работа с API, ai - работа с Ollama/LLM, email - отправка почты; оркестрирующие скрипты - в корне run
- Имя файла скрипта: префикс по домену (db_, api_, ai_, email_) + описание действия, например `db_init.py`, `api_sync.py`, `ai_ollama_check.py`, `email_test_send.py`
- Для проектов с БД обязательны скрипты в run/db/: db_check_connection.py, db_check_schema.py, db_init.py
- Для проектов с внешними API обязательны скрипты в run/api/: api_check_connection.py, api_test_requests.py
- Для проектов с Ollama обязательны скрипты в run/ai/: ai_ollama_check.py, ai_ollama_pull_models.py
- Для проектов с рассылкой почты обязателен скрипт в run/email/: email_test_send.py

**Оркестрация:** если в проекте предусмотрена общая оркестрация (единая точка входа для всего пайплайна), допускается создание файла `main.py` в корне проекта.

## 01.02. Конфигурация

*Обязательно.*

- Хранить все настраиваемые параметры проекта в config/config.ini
- Не хардкодить в коде значения, которые пользователь может менять (пути, порты, имена и т.п.)
- Читать настройки из config/config.ini при запуске
- ConfigParser возвращает все значения как строки; использовать типизированные методы: `config.getint()`, `config.getfloat()`, `config.getboolean()` - не приводить типы вручную через `int()` и т.п.
- Секция `[app]` в config.ini обязательна; содержит параметры `name` (название приложения) и `version` (версия, SemVer). См. раздел 01.07
- Секция `[app]` может содержать параметр `language` - язык интерфейса (подробнее в разделе 04.01)

## 01.03. Пути к файлам проекта

*Обязательно.*

- PROJECT_ROOT вычисляется один раз - в `src/config.py`: `Path(__file__).resolve().parents[1]`
  (файл всегда находится в `src/`, поэтому `parents[1]` - всегда корень проекта)
- Все остальные скрипты получают PROJECT_ROOT через импорт: `from src.config import config, PROJECT_ROOT`
  Не вычислять PROJECT_ROOT повторно в каждом скрипте
- Все пути строить через pathlib относительно PROJECT_ROOT: `PROJECT_ROOT / "log" / "file.log"`
  Не использовать строковую конкатенацию путей и `os.path.abspath()`
- Не полагаться на текущую рабочую директорию (cwd); скрипт должен корректно работать при запуске из любой папки
- `src/config.py` также экспортирует `APP_NAME` и `APP_VERSION` из секции `[app]` конфига (см. раздел 01.07)

## 01.04. Валидация при запуске

*Обязательно.*

Каждый скрипт перед выполнением основной логики проверяет готовность окружения. Цель - обнаружить проблемы до начала работы и выдать понятное сообщение вместо непредсказуемого сбоя в процессе.

**Проверять при запуске:**
- Наличие обязательных параметров в config.ini и .env (без значений - сразу ошибка с указанием имени параметра)
- Корректность путей: существование папок log/, data/ и других, используемых скриптом
- Доступность внешних ресурсов: БД, API, Ollama, почта (SMTP) - только если скрипт с ними работает (см. разделы 02.01-02.03, 02.06)
- Наличие обязательных файлов: schema.sql для db-скриптов, config.ini и т.п.

**Принцип fail fast:**
- Все проверки выполняются до начала основной логики
- При обнаружении проблемы - сразу выход с понятным сообщением (через t(), см. раздел 04.01) и exit(1)
- Не продолжать выполнение, если обязательные условия не выполнены
- Несколько проблем - выводить все одним блоком, не останавливаться на первой (агент может исправить все за один проход)

**Порядок проверок:**
1. Чтение конфигурации (config.ini, .env)
2. Валидация обязательных параметров
3. Проверка путей и файлов
4. Проверка внешних подключений (если применимо)

## 01.06. Виртуальное окружение и Python-версия

*Обязательно.*

- Хранить окружение внутри проекта: папка .venv в корне проекта
- Создание: `python3 -m venv --copies .venv` (флаг `--copies` обеспечивает реальные файлы вместо симлинков; расширение VS Code Python Environments не принимает цепочки симлинков как валидный интерпретатор)
- Активация: source .venv/bin/activate
- Установка зависимостей: pip install -r requirements.txt
- Исключить .venv из git: добавить `.venv/` в .gitignore
- Фиксировать версию Python в файле .python-version в корне проекта; использовать Python 3.14
- Зеркало pip: pypi.org может быть недоступен напрямую (несовместимость TLS, сетевые ограничения).
  Перед установкой зависимостей убедиться, что ~/.config/pip/pip.conf указывает на рабочее зеркало.
  Агент НЕ хардкодит конкретное зеркало - он подбирает самое быстрое и доступное на текущей машине
- Порядок подбора зеркала:
  1. Проверить наличие ~/.config/pip/pip.conf и работоспособность указанного в нем index-url
     (быстрый HEAD/простой запрос). Если текущее зеркало отвечает быстро - оставить его без изменений
  2. Если зеркало отсутствует или не отвечает - агент прозванивает кандидатов из списка ниже,
     измеряет время отклика каждого и выбирает самый быстрый доступный
  3. Записать выбранное зеркало в ~/.config/pip/pip.conf
- Примерные зеркала для РФ (агент проверяет доступность и скорость каждого, выбирает лучший):
  - https://pypi.ru/simple/ - российское публичное зеркало
  - https://mirror.yandex.ru/pypi/simple/ - зеркало Яндекса
  - https://mirrors.aliyun.com/pypi/simple/ - зеркало Aliyun (стабильно работает из РФ)
  - https://pypi.org/simple/ - официальный источник (проверять последним, часто недоступен)
- Шаблон создания/обновления файла (подставить выбранное зеркало):
  ```
  mkdir -p ~/.config/pip && cat > ~/.config/pip/pip.conf << 'EOF'
  [global]
  index-url = <выбранное_зеркало>
  EOF
  ```
- Агент сообщает пользователю, какое зеркало выбрано и почему (время отклика, причина выбора)

## 01.07. Версионность приложения

*Обязательно.*

Каждый проект имеет название и версию приложения. Эти значения используются для идентификации проекта в логах, консоли и при развертывании.

**Название приложения:**
- Агент запрашивает название у пользователя явно при инициализации проекта (раздел 07.03)
- Название хранится в секции `[app]` файла `config/config.ini`: параметр `name`
- Название может содержать буквы, цифры, пробелы, дефисы - в человекочитаемом виде (например: `Мой инструмент`, `Data Processor`)

**Методология определения названия:**

Назначение названия - идентификация проекта пользователем. Название отображается в логах и консоли при каждом запуске (строка `{APP_NAME} v{APP_VERSION}`).

Правила:
- Агент не придумывает название самостоятельно - только из ответа пользователя
- Если пользователь не указал название явно - агент задает вопрос: "Укажите название приложения" - до начала создания структуры проекта
- Название задается один раз при инициализации и не меняется в течение жизни проекта
- Название не связано с именем папки репозитория, именем пакета в `pyproject.toml` или именем CLI-команды - это разные сущности:
  - Название приложения (`config.ini [app] name`) - человекочитаемое, для отображения пользователю
  - Имя пакета (`pyproject.toml name`) - техническое, для pip/uv (строчные буквы, дефисы)
  - CLI-команда (`pyproject.toml [project.scripts]`) - команда shell
- Допускается изменение названия пользователем в `config.ini` вручную в любой момент; агент не предлагает изменить название при коммитах

**Версия приложения:**
- Хранится в секции `[app]` файла `config/config.ini`: параметр `version`
- Начальная версия при создании проекта: `0.1.0`
- Формат версии: семантическое версионирование (SemVer) - `MAJOR.MINOR.PATCH`

**Методология определения версии (SemVer):**

- **PATCH** (X.Y.Z → X.Y.Z+1): исправление ошибок, мелкие доработки без нового функционала
- **MINOR** (X.Y.Z → X.Y+1.0): новый функционал, новые возможности, обратно совместимые изменения
- **MAJOR** (X.Y.Z → X+1.0.0): ломающие изменения - несовместимость с предыдущей версией, изменение интерфейса или поведения

**Автоматическое предложение обновления версии при коммите:**

При каждом коммите агент анализирует тип коммита (раздел 06.01) и предлагает обновить версию в `config/config.ini`:

| Тип коммита | Действие с версией |
|-------------|-------------------|
| `feat` | MINOR |
| `fix` | PATCH |
| `refactor` | PATCH (если поведение не изменилось) или MINOR (если появились новые возможности) |
| `chore` | PATCH |
| `docs` | не обновлять |
| `test` | не обновлять |
| `style` | не обновлять |

Порядок:
1. Агент определяет тип коммита
2. Если тип подразумевает обновление версии - агент предлагает пользователю новую версию
3. При согласии - агент обновляет `version` в секции `[app]` файла `config/config.ini` и включает это изменение в текущий коммит
4. При отказе - версия остается без изменений
5. MAJOR-обновление агент не предлагает автоматически - только по явному запросу пользователя

**Экспорт в config.py:**
- `src/config.py` экспортирует `APP_NAME` (str) и `APP_VERSION` (str) - значения из секции `[app]`
- При отсутствии значения `name` - пустая строка; при отсутствии `version` - строка `dev`

**Вывод при запуске:**
- Каждый entry-point скрипт (в `run/`, `main.py`) выводит название и версию одним из первых действий, сразу после инициализации логера:

- Вывод названия и версии - перед основной логикой скрипта, до любых действий с данными

**Для проектов с деплоем** (по стандартам `project_deploy_standards_lin.md` / `project_deploy_standards_win.md`):
- `config.ini` секция `[app]` - единый источник `name` и `version` для отображения во всех режимах (разработки и установленном)
- `pyproject.toml` поле `version` - источник для метаданных пакета (pip/uv), не используется для отображения

- Установочный скрипт синхронизирует `name` и `version` из репо-конфига в прод-конфиг при обновлении (подробнее - в стандартах деплоя)

## 02.04. Retry для внешних подключений

*Опционально - только если проект предусматривает внешние подключения.*

Все операции с внешними ресурсами (БД, API, Ollama, почта) подвержены временным сбоям: таймауты, обрывы соединения, временная недоступность. Скрипты должны обрабатывать такие сбои через механизм повторных попыток.

**Параметры (в config/config.ini):**
- `retry_count` - количество повторных попыток (по умолчанию 3)
- `retry_delay` - начальная задержка между попытками в секундах (по умолчанию 2)

**Правила:**
- Использовать экспоненциальный backoff: каждая следующая попытка с задержкой `retry_delay * 2^attempt`
- Максимальная задержка - 30 секунд
- Логировать каждую попытку: номер попытки, задержка, причина повтора
- После исчерпания попыток - логировать ошибку и exit(1)
- Не применять retry для ошибок аутентификации (401/403) и ошибок валидации (400/422) - только для таймаутов и сетевых ошибок (5xx, ConnectionError, TimeoutError)
- Для БД: retry при OperationalError, TimeoutError; не retry при ProgrammingError (ошибка в SQL)

**Реализация:** общая функция `retry()` в `src/` для всех типов подключений.

## 02.05. GitHub

*Опционально - только если проект предусматривает публикацию на GitHub.*

**SSH-ключ**

Не искать файл ключа по конкретному имени. Имя может быть нестандартным (`id_ed25519_work_github`, `id_rsa_company_github` и т.п.). Сканировать `~/.ssh/` по паттерну `*github*.pub`; брать первый найденный файл. Приватный ключ - тот же файл без суффикса `.pub`:

```bash
KEY_PUB=$(ls ~/.ssh/*github*.pub 2>/dev/null | head -1)

if [ -z "$KEY_PUB" ]; then
    echo "[!] SSH-ключ для GitHub не найден в ~/.ssh/ (паттерн: *github*.pub)"
    exit 1
fi

KEY_PATH="${KEY_PUB%.pub}"
```

**Создание ключа (если не найден)**

Если ключ не обнаружен - сообщить пользователю и вывести команды для создания. Имя файла строить по паттерну `id_ed25519_{hostname}_github`, чтобы ключ автоматически обнаруживался при следующем запуске:

```
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_$(hostname -s)_github -C "your@email"
```

Затем добавить публичный ключ в аккаунт GitHub: Settings → SSH and GPG keys → New SSH key, вставить содержимое файла `~/.ssh/id_ed25519_$(hostname -s)_github.pub`.

Паттерн `id_ed25519_{hostname}_github` гарантирует, что glob `*github*.pub` найдет ключ при следующем вызове.

**Проверка соединения**

Передавать найденный ключ явно через флаг `-i`. Не полагаться на `~/.ssh/config` - при прямом подключении к `ssh.github.com` конфиг с `Host github.com` на это подключение не распространяется:

```bash
ssh -i "$KEY_PATH" -T -p 443 git@ssh.github.com
```

Ожидаемый вывод: `Hi <username>! You've successfully authenticated...`

**Git-операции по сети**

Все git-команды с сетевым доступом (`fetch`, `push`, `clone`) выполнять с явно заданным ключом через `GIT_SSH_COMMAND`. Устанавливать до первой git-команды, требующей сетевого доступа:

```bash
export GIT_SSH_COMMAND="ssh -i $KEY_PATH -o BatchMode=yes -o StrictHostKeyChecking=no"
```

## 03.01. SQL и DDL

*Опционально - только если проект предусматривает работу с БД.*

**Правила работы с SQL (если в проекте есть БД):**

- Список ожидаемых таблиц и колонок определяется из схемы; проверка соответствия - через db_check_schema

## 03.02. Скрипты run/db

*Опционально - только если проект предусматривает работу с БД.*

Для проектов с БД создавать в run/db/ три скрипта:


| Скрипт                 | Назначение                                                           |
| ---------------------- | -------------------------------------------------------------------- |
| db_check_connection.py | Проверка соединения с БД                                             |
| db_check_schema.py     | Проверка наличия в БД нужных таблиц в соответствии со схемой из sql/ |
| db_init.py             | Создание или пересоздание таблиц по схеме; очистка при соответствии  |


**Логика db_check_connection:** подключение к БД по настройкам (config.ini / .env). При успехе - `[i] Соединение с БД установлено`. При ошибке - `[!]` и exit(1).

**Логика db_check_schema:** читает схему из sql/ (файлы .sql с CREATE TABLE), проверяет наличие в БД всех таблиц и всех колонок каждой таблицы. Если схема соответствует - `[i] Схема соответствует`. Если не соответствует - `[*] Схема не соответствует. Рекомендуется выполнить db_init`.

**Логика db_init:** проверяет схему (как db_check_schema). Если схема соответствует: запрос «Очистить таблицы? 1 - да, 0 - нет»; при 1 - очистка через TRUNCATE (быстро, сбрасывает автоинкремент), при 0 - выход. Если схема не соответствует: запрос «Пересоздать таблицы по схеме? 1 - да, 0 - нет»; при 0 - выход; при 1 - для каждой таблицы из схемы: если таблица не существует - выполнить CREATE из sql/; если таблица существует, но структура отличается - выполнить DROP TABLE, затем CREATE из sql/.

## 03.05. Идемпотентность и атомарность

*Обязательно.*

Скрипты проекта должны быть безопасны для повторного запуска. Сбой в середине выполнения не должен оставлять данные в неконсистентном состоянии.

**Идемпотентность:**
- Повторный запуск скрипта должен приводить к тому же результату, что и первый запуск
- Скрипты run/db: db_init корректно обрабатывает частично созданные таблицы (см. раздел 03.02)
- Скрипты обработки данных: не создавать дубли при повторном запуске; использовать UPSERT, проверку существования записи или уникальные индексы
- Загрузка данных: пропускать уже загруженные записи (по дате, ID или другому маркеру)

**Атомарность записи файлов:**
- Для критичных файлов (конфиг, данные, состояние) использовать паттерн write-to-temp + rename:
  ```python
  temp_path = target_path.with_suffix(".tmp")
  temp_path.write_text(content, encoding="utf-8")
  temp_path.replace(target_path)
  ```
- replace() атомарен на обеих ОС (на Windows - при замене на ту же файловую систему)
- Не записывать данные напрямую в целевой файл - при сбое (питание, Ctrl+C) файл будет поврежден

**Атомарность БД:**
- Все операции, изменяющие несколько таблиц или строк - внутри одной транзакции
- Не коммитить промежуточные результаты по одной записи - собрать batch и выполнить одну транзакцию
- При ошибке в середине - откат всей транзакции (rollback)

## 03.08. Кросс-платформенные пути

*Обязательно.*

Скрипты могут запускаться на Windows и Linux. Все операции с путями должны работать на обеих ОС автоматически.

- Использовать `pathlib.Path` для всех путей - он автоматически применяет разделитель, принятый в ОС (`\` на Windows, `/` на Linux)
- Не конкатенировать пути через строки (`"log/" + name`) и не использовать `os.path.join()` со строками-частями
- Не использовать захардкоженные разделители (`/` или `\`) в путях - только методы `Path`: оператор `/` или `.joinpath()`
- Пути из `config.ini` или `.env` оборачивать в `Path()` при чтении: `log_dir = Path(config.get("paths", "log_dir"))`
- Пути в `config.ini` записывать с прямым слешем (`log/app.log`) - `pathlib.Path` корректно распознает их на обеих ОС
- Для создания директорий: `path.mkdir(parents=True, exist_ok=True)` - работает на обеих ОС
- Для определения ОС при необходимости: `import sys; sys.platform` (`"win32"` на Windows, `"linux"` на Linux) - использовать только если логика принципиально различается по ОС

## 04.01. Мультиязычность (i18n)

*Обязательно.*

Все выводы приложения (пользовательские сообщения, лог-сообщения, элементы меню, ошибки, промпты) проходят через единую функцию `t(key, **kwargs)` из модуля `src/localization.py`. В коде не должно быть строковых литералов на естественном языке - ни в logger.info(), ни в print(), ни в raise, ни в любых других местах. Исключение: файлы переводов и комментарии в коде.

**Принципы:**
- Все текстовые строки вынесены из кода в хранилище переводов (БД или JSON)
- Доступ к переводам - через единую функцию `t(key, **kwargs)`
- Каждый ключ имеет перевод на всех поддерживаемых языках (нет частичной локализации)
- Fallback-цепочка: текущий язык -> язык по умолчанию -> сигнатура `!KEY_NAME!`
- Поддержка плюрализации для языков с разными формами множественного числа

**Хранилище переводов:**
- Для проектов с SQLite - таблица `translations` (key TEXT + lang TEXT composite PK, value TEXT). Добавление нового языка не требует ALTER TABLE
- Для проектов без БД - JSON-файл `config/translations.json` с секциями meta (default_lang, languages) и translations

**Модуль src/localization.py:**
- `init(language=None)` - определение языка, загрузка переводов, сборка словаря
- `t(key, *, count=None, **kwargs)` - единственная точка доступа к переводам. Поддержка интерполяции через str.format() и плюрализации через параметр count
- `reload(language=None)` - перезагрузка словаря (смена языка без перезапуска)
- Кэш в памяти: словарь `{key: text}` на текущем языке. Загрузка при первом импорте (early initialization)

**Именование ключей:**
Иерархическая схема с точкой-разделителем. Пространства: `menu.*`, `msg.*`, `error.*`, `title.*`, `prompt.*`, `label.*`, `help.*`. Латинские буквы, точка, нижнее подчеркивание. Ключи стабильны: не переименовывать. Для плюральных форм - суффикс: `msg.items_found.one`, `msg.items_found.many`.

**Плюрализация:**
Суффиксы ключей: `.zero`, `.one`, `.few`, `.many`. Русский: 4 формы. Английский: 2 формы (one/many). Выбор формы по правилам языка автоматически при указании count.

**Определение языка (приоритет):**
1. Параметр `language` в секции `[app]` config/config.ini
2. Переменная окружения `LANGUAGE` (POSIX)
3. Системная локаль
4. `en` по умолчанию

**Интеграция с логированием:**
Все вызовы logger.info(), logger.error() и т.д. используют t():
```python
logger.info(t("msg.load_started"))
logger.error(t("error.connection_failed", detail=str(e)))
```

**Интеграция с меню:**
Пункты меню хранят ключи переводов, не текст. Рендер через t() в момент отображения (см. раздел 04.06).

**Заполнение начальных переводов:**
При инициализации проекта (db_init.py) - заполнить таблицу/файл начальными переводами для всех поддерживаемых языков.

## 04.02. Логирование

*Обязательно.*

- Использовать общий модуль `src/logger.py` для всех скриптов; загрузку config и .env вынести в `src/config.py` или аналог
- Типовой паттерн инициализации логгера в начале каждого скрипта:
  ```python
  from src.config import PROJECT_ROOT, APP_NAME, APP_VERSION
  from src.logger import get_logger
  from src.localization import t
  logger = get_logger(__name__, log_dir=PROJECT_ROOT / "log")
  logger.info(t("msg.app_started", name=APP_NAME, version=APP_VERSION))
  ```
- Всегда создавать лог-файл для каждого скрипта из run и для main.py; делать это по умолчанию, даже если пользователь не просил явно
- Создавать лог-файлы в папке log
- Имя файла: `log_[имя_скрипта]_[timestamp].log`, где timestamp - момент запуска
- Один запуск скрипта - один лог-файл
- Логировать все, что выводится в консоль, всегда и без исключений; вывод в лог и в консоль идентичен по содержанию и формату. Опциональное исключение - индикация прогресса (раздел 04.03.01): промежуточные строки прогресса выводятся только в консоль и не дублируются в лог-файл; итоговая строка по завершении операции пишется через logger.info() и попадает в лог
- Использовать единый логгер проекта для вывода в файл и консоль; не вызывать print()
- Сообщения в логе: краткие, без лишних слов, без эмодзи

**Режимы форматирования:**

Единый модуль `src/logger.py` поддерживает два режима через параметр `menu_mode` в `get_logger(name, log_dir, menu_mode=False)`:

- `menu_mode=False` (по умолчанию) - для обычных скриптов (run/, пакетная обработка): формат `YYYY-MM-DD HH:MM:SS [маркер] сообщение`
- `menu_mode=True` - для проектов с интерактивным меню: формат `сообщение` (без временной метки и маркера уровня)

Оба режима пишут идентично в файл и консоль - разница только в наличии или отсутствии префикса. `_MarkerFormatter` при `menu_mode=True` опускает timestamp и маркер. Выбор режима определяется типом скрипта: entry-point с меню использует `menu_mode=True`; скрипты в `run/` используют `menu_mode=False`.

## 04.03. Форматирование вывода в лог и консоль

*Обязательно.*

### 04.03.01. Индикация прогресса в консоли

*Опционально - только если проект предусматривает длительные операции с обратной связью.*

Для операций, где требуется показывать прогресс в реальном времени (обработка файлов, загрузка данных, длительные циклы), используется перезапись одной строки через символ возврата каретки `\r`. Вывод идет напрямую в `sys.stdout`, минуя логгер - строки прогресса не дублируются в лог-файл. В лог-файл попадает только итоговая строка через `logger.info()`.

В проекте создается модуль `src/progress.py`:

```python
import sys

def update_progress(line: str) -> None:
    """Перезаписать текущую строку в консоли."""
    sys.stdout.write(f"\r{line}")
    sys.stdout.flush()

def clear_progress() -> None:
    """Очистить строку прогресса (после завершения)."""
    sys.stdout.write("\r\033[K")
    sys.stdout.flush()
```

`\033[K` - ANSI-escape «стереть до конца строки» (ED - Erase in Line). Корректно работает на Linux, macOS и Windows 10+; для Python 3.14 более старые Windows не актуальны.

**Пример использования:**

```python
from src.progress import update_progress, clear_progress
from src.logger import get_logger

total = 5047
for i in range(total + 1):
    pct = i / total * 100
    update_progress(f"Обработка... {pct:5.1f}%  ({i}/{total})")

clear_progress()
logger.info(f"Обработано {total} файлов")
```

Вывод в консоли (одна меняющаяся строка):

```
Обработка...  45.2%  (2285/5047)
```

**Правила:**

- `update_progress()` не вызывает перевод строки - каждый следующий вызов перезаписывает ту же строку
- После завершения операции обязателен вызов `clear_progress()` - иначе строка прогресса останется в консоли
- Прогресс не пишется в лог-файл: `sys.stdout.write()` минует логгер
- Итоговый результат всегда через `logger.info()` - он попадает и в консоль, и в лог
- К тексту прогресса применяются общие требования раздела 04.05: дефис вместо тире, без многоточий, без эмодзи, без точки в конце
- При прерывании скрипта (Ctrl+C) обработчик сигнала (раздел 05.03) должен вызвать `clear_progress()` перед завершением
- Использование `sys.stdout.write()` в этом модуле - допустимое исключение из общего правила «не вызывать print()» (аналогично `src/menu.py` в разделе 04.06)
- Зависимости: не требуются - только `sys` из стандартной библиотеки

### 04.03.02. Формат строки

*Обязательно.*

- Формат строки зависит от режима логгера (раздел 04.02):
  - Обычный режим (`menu_mode=False`): `YYYY-MM-DD HH:MM:SS [уровень] сообщение`, например: `2025-03-02 14:30:15 [i] Загрузка данных начата`
  - Режим меню (`menu_mode=True`): `сообщение` (без префикса), например: `Загрузка данных начата`
- Timestamp в формате ISO 8601 или `YYYY-MM-DD HH:MM:SS` (только в обычном режиме)
- Уровень сообщения указывать маркером в квадратных скобках (из раздела 04.04: [!], [i], [*]) в зависимости от типа
- Маркер уровня вставляется автоматически форматтером логгера (`_MarkerFormatter` в `src/logger.py`); в текст сообщения маркер не включать: `logger.info(t("msg.load_started"))`, а не `logger.info(t("msg.load_started_marker"))`

### 04.03.03. Общие правила оформления

*Обязательно.*

- Одна строка - одно сообщение; при переносе длинного текста использовать отступ для продолжения
- По возможности сокращать количество строк: объединять связанные данные в одну строку, если это сохраняет смысл и читаемость
- Вывод в лог и консоль идентичен; не дублировать одно и то же разным форматом
- Не использовать цветовое выделение и эмодзи
- Не ставить точку в конце сообщений: `Загрузка завершена`, а не `Загрузка завершена.`
- Не использовать строки-разделители (`=====`, `-----` и подобные) - ни в логе, ни в консоли

### 04.03.04. Структурирование вывода

*Обязательно.*

- Логические блоки вывода (этапы выполнения, результаты операций, группы связанных сообщений) визуально отделяются друг от друга пустой строкой: `\n\n` (ровно одна пустая строка между блоками). Это позволяет структурировать вывод без использования строк-разделителей, которые запрещены

### 04.03.05. Таблицы

*Обязательно.*

- Все таблицы в выводе (лог и консоль) формируются только через библиотеку `tabulate`. Допустимые форматы: `grid`, `simple`, `pipe`, `outline`. Библиотека минималистична, не требует сторонних зависимостей, обеспечивает единообразное выравнивание колонок. Иные способы вывода таблиц (ручная сборка строк, пробельное выравнивание, f-строки) не допускаются
- Заголовки столбцов таблиц выравнивать по левому краю. Выравнивание настраивается через параметры `tabulate` (например, `colalign=("left", ...)`)
- Числовые значения в ячейках таблиц выравнивать по правому краю. Выравнивание настраивается через `colalign`: столбцы с текстом - `"left"`, столбцы с числами - `"right"`
- Каждой таблице в выводе должен предшествовать осмысленный заголовок (caption) - отдельная строка с текстом, описывающим содержимое таблицы. Заголовок выводится на строке непосредственно перед таблицей, отделяется от нее одним переводом строки. Заголовок проходит через `t()`

### 04.03.06. Списки

*Обязательно.*

- Для вывода списков однотипных элементов используется формат: заголовок списка (первая строка, без префикса), затем каждый элемент с префиксом `"- "` (дефис и пробел). Заголовок и текст элементов проходят через `t()`. Пример структуры:
  - Заголовок: `{t("msg.list_title")}` - вывод через `logger.info()`
  - Элементы: `- {t("msg.list_item")}` - каждый элемент выводится отдельным вызовом `logger.info()` с префиксом `"- "`
  Между заголовком и элементами пустая строка не ставится

### 04.03.07. Сводки

*Обязательно.*

- Если в одной строке требуется перечислить несколько разнородных параметров или счетчиков, используется формат: `{t("msg.summary", param1=value1, param2=value2, ...)}` - сначала текст сводки, затем через запятую и пробел пары `ключ: значение`. Никаких дополнительных разделителей (скобок, дефисов, вертикальных черт) не используется

## 04.05. Требования к написанию текста

*Обязательно.*

Применять ко всем генерируемым текстам: логи, консоль, документация, комментарии, сообщения пользователю.

- Все текстовые строки для вывода (лог, консоль, меню) - через функцию t() из src/localization.py (раздел 04.01). Не использовать строковые литералы на естественном языке в коде
- Язык комментариев в коде и документации: русский
- Избегать эмоциональных и оценочных формулировок: восклицаний, призывов, выражений типа «важно», «критично», «невероятно» и т.п.
- Не использовать заглавные буквы для усиления акцента (например, СРОЧНО, ВНИМАНИЕ). Для выделения использовать обычный регистр (Срочно) или маркеры из раздела 04.04
- Соблюдать правила русского языка: орфография, пунктуация, согласование, управление
- Во всех генерируемых текстах (код, комментарии, документация, логи) использовать русские е и Е без точки над буквой (не использовать символы U+0451 и U+0401 в русских словах; только U+0435 и U+0415), сохраняя исходный регистр
- Весь технический нейминг (переменные, функции, модули, имена файлов, идентификаторы) - строго на латинице. Подробнее о стилях именования см. в разделе 03.06
- При использовании цифр в неймингах применять лидирующие нули для выравнивания текста (например: item_001, step_02, batch_003)
- Использовать дефис `-` (U+002D) вместо длинного тире `—` (U+2014) и короткого тире `–` (U+2013) во всех генерируемых текстах: логи, консоль, документация, комментарии, сообщения пользователю
- Не использовать многоточия (три точки `...` и единый символ `…` (U+2026)) нигде: ни в логах, ни в консоли, ни в документации, ни в комментариях, ни в сообщениях пользователю
- При открытии файлов через `open()` всегда явно указывать `encoding="utf-8"`
- Не начинать новые строки текста с пробелов (ни с одного, ни с нескольких). Лидирующие пробелы в начале абзацев и строк недопустимы

## 04.06. Интерактивные меню

*Опционально - только если проект предусматривает интерактивные меню.*

Если в проекте есть интерактивные меню (выбор действия пользователем), соблюдать правила:

- Не выводить элементы меню через логгер - логгер не должен использоваться для отображения пунктов меню и приглашения ввода
- Для реализации меню создавать отдельный модуль `src/menu.py` (или аналогичный), который отвечает только за отображение меню и получение выбора пользователя
- Каждое меню обязано иметь заголовок - первая строка меню, обрамленный дефисами
- Перед заголовком меню всегда выводится пустая строка - для визуального отделения меню от предшествующего вывода. Пустая строка ставится перед каждым выводом меню: при первом запуске (после строки запуска приложения), при перерисовке после выполнения действия, при возврате из подменю
- Заголовок формируется как цепочка breadcrumb от корневого меню до текущего раздела через разделитель ` > ` (пробел, знак больше, пробел)
- Заголовок главного меню (корень): `--- {t("title.main_menu")} ---`
- Заголовок подменю первого уровня: `--- {t("title.main_menu")} > {t("title.section_name")} ---`
- Заголовок подменю второго уровня: `--- {t("title.main_menu")} > {t("title.section_name")} > {t("title.subsection_name")} ---`
- Каждый следующий уровень вложенности добавляет к цепочке ` > ` и название раздела

**Модель навигации и формирование breadcrumb.**

- Breadcrumb в заголовке меню отражает путь навигации - последовательность выбранных пользователем пунктов от корневого меню до текущего раздела
- Каждый сегмент breadcrumb соответствует реальному пункту меню, по которому выполнен переход: в цепочку попадают только те пункты, по которым пользователь фактически перемещался
- В breadcrumb не включаются заголовки содержимого (надписи над таблицами, например «Доступные хосты») и универсальные подписи (например «Действия»): они не являются пунктами навигации
- Правило «один переход - один сегмент»: при возврате на уровень вверх соответствующий сегмент удаляется из цепочки, при переходе вглубь - добавляется
- Формат пункта меню: `[N] {t("menu.action_name")}`, где N - номер или буквенный идентификатор пункта
- В главном меню номер 0 резервирован для выхода из скрипта: `[0] {t("menu.exit")}`. Номер 9 в главном меню отсутствует
- В подменю второго уровня (первое подменю) номер 0 резервирован для возврата в главное меню: `[0] {t("menu.main_menu")}`. Номер 9 доступен для функциональных разделов; кнопка «Назад» на этом уровне не нужна, так как возврат на шаг вверх и возврат в главное меню приводят к одному и тому же результату
- В подменю третьего уровня и глубже номер 0 резервирован для возврата в главное меню: `[0] {t("menu.main_menu")}`. Номер 9 резервирован для возврата на шаг назад: `[9] {t("menu.back")}`

Зарезервированные номера используют фиксированные ключи переводов. Значения переводов для этих ключей зафиксированы и не должны различаться между проектами:

| Номер | Уровень | Ключ | Русский | English |
|-------|---------|------|---------|---------|
| 0 | Главное меню | `menu.exit` | Выход | Exit |
| 0 | Подменю (уровень 2+) | `menu.main_menu` | Главное меню | Main menu |
| 9 | Подменю (уровень 3+) | `menu.back` | Назад | Back |
- Последняя строка меню - приглашение ввода: `[>]`
- После ввода пользователя всегда выводится пустая строка - для визуального отделения ввода от последующего вывода (результатов действия, сообщения об ошибке, нового меню). Пустая строка ставится после любого ответа пользователя в приглашении `[>]`, включая выбор пункта и ввод данных
- Действия пользователя с меню подлежат обязательному логированию как информационные события. Логированию подлежат только действия пользователя с меню - выбор пункта, ввод данных, навигация. Сами пункты меню, заголовки, приглашения ввода и оформительские элементы логировать запрещено
- Выбор пункта меню записывается в лог в следующем формате:
  1. Строка ввода пользователя: `[>] N` (N - выбранный номер пункта)
  2. Строка подтверждения выбора: `Выбран пункт [N]` (через `t("msg.user_selected", number=N)`)
  3. Пустая строка после подтверждения - для визуального отделения от последующего вывода
- Иной формат записи выбора пункта меню не допускается
- Логгер для entry-point с меню инициализируется с `menu_mode=True` (раздел 04.02): сообщения в консоли и лог-файле выводятся без временных меток и маркеров уровня
- Информационные сообщения, порожденные действиями пользователя (результаты сканирования, статус проверки, ошибки), пишутся через тот же логгер - в консоль и файл без префикса
- Блоки результатов визуально отделяются от меню пустой строкой: один перевод строки перед блоком и один после. Это обеспечивает читаемость без разделителей `-----`, которые запрещены (раздел 04.03)

**Пример правильного вывода главного меню:**
```

--- {t("title.main_menu")} ---
[1] {t("menu.main.scan")}
[2] {t("menu.main.settings")}
[3] {t("menu.main.help")}
[0] {t("menu.exit")}
[>]
```

**Пример правильного вывода подменю второго уровня (первое подменю, без кнопки 9):**
```

--- {t("title.main_menu")} > {t("title.settings")} ---
[1] {t("menu.settings.network")}
[2] {t("menu.settings.interface")}
[0] {t("menu.main_menu")}
[>]
```

**Пример правильного вывода подменю третьего уровня (с кнопкой 9):**
```
--- {t("title.main_menu")} > {t("title.settings")} > {t("title.network")} ---
[1] {t("menu.network.check")}
[2] {t("menu.network.proxy")}
[9] {t("menu.back")}
[0] {t("menu.main_menu")}
[>]
```

**Пример вывода результата действия с отбивкой пустой строкой:**
```
[>] 1
Выбран пункт [1]

Найдено: 42 ссылки
Обработано за 3.2 сек

--- {t("title.main_menu")} ---
[1] {t("menu.main.scan")}
[2] {t("menu.main.settings")}
[3] {t("menu.main.help")}
[0] {t("menu.exit")}
[>]
```

Внутренний скрипт меню может использовать `print()` для отображения пунктов - это допустимое исключение из общего правила "не использовать print()"

## 04.07. Формулировка сообщений об ошибках и терминология

*Обязательно.*

**Сообщения об ошибках для пользователя:**

- Сообщения об ошибках в логе и консоли формулировать кратко, на понятном языке, без технического жаргона - одна строка, суть ошибки
- Не выводить пользователю сырой stderr/stdout системных утилит; вместо этого анализировать код возврата и текст ошибки, и выдавать человекопонятное описание
- Для известных кодов и паттернов ошибок - заменять технический текст на понятное сообщение; для неизвестных - выводить исходный текст ошибки
- Не использовать внутренние технические термины (имена системных файлов, конфигураций, протоколов), если они не входят в рабочий лексикон пользователя проекта

**Единообразие терминологии:**

- Во всех пользовательских сообщениях (логи, консоль, меню) использовать единый набор терминов для одних и тех же сущностей; не смешивать синонимы

## 04.08. Графический интерфейс (GUI)

*Опционально - только если проект предусматривает графический интерфейс.*

Графический интерфейс - один из возможных режимов точки входа для пользователя, наряду с CLI-скриптами (`run/`) и текстовым меню (раздел 04.06). Проект может использовать любой режим или их комбинацию; GUI не отменяет необходимости CLI-скриптов проверки и обслуживания (`run/db`, `run/api`, `run/ai`), если соответствующие домены присутствуют.

**Точка входа и запуск:**
- Запуск GUI выполняется из `main.py` - он создает и отображает главное окно
- Сразу после инициализации логера `main.py` выводит название и версию через `logger.info(t("msg.app_started", name=APP_NAME, version=APP_VERSION))` - в соответствии с разделами 01.07 и 04.02
- Заголовок главного окна: `{APP_NAME} v{APP_VERSION}`
- Перед стартом окна выполняется стартовая валидация окружения (раздел 01.04): при критических проблемах - вывод сообщений через `t()` и `exit(1)` вместо открытия окна

**Технология:**
- Рекомендуемая библиотека по умолчанию - `tkinter` (стандартная библиотека Python, кросс-платформенная: Windows и Linux, не требует внешних зависимостей). Это соответствует принципу предпочитать стандартную библиотеку (раздел 01.05)
- Прочие библиотеки (`customtkinter`, `PySide6`/`PyQt`, `textual` и подобные) допустимы - с обоснованием необходимости в `requirements.txt` и при условии кросс-платформенности (Windows и Linux, раздел 03.08)
- Не использовать библиотеки, привязанные к одной ОС

**Расположение кода:**
- Код графического интерфейса размещается в подпапке `src/gui/`: окна, виджеты, компоновки, обработчики событий
- `src/gui/` - аналог модуля `src/menu.py` (раздел 04.06) для текстовых меню, но для графического интерфейса
- Бизнес-логика остается в `src/` и `run/`; модули `src/gui/` отвечают только за отображение и обработку взаимодействия с пользователем

**Текст интерфейса:**
- Весь текст виджетов (заголовки окон, надписи, кнопки, сообщения, подсказки) - только через `t()` из `src/localization.py` (раздел 04.01)
- Строковые литералы на естественном языке в коде виджетов не допускаются
- Переключение языка интерфейса - через `reload()` модуля локализации (раздел 04.01) без перезапуска приложения

**Логирование:**
- Действия пользователя (нажатия кнопок, выбор пунктов, ввод данных, навигация) логируются как информационные события - по аналогии с разделом 04.06
- Сам рендер окна, перерисовка виджетов и оформительские элементы через логгер не дублируются
- Логгер инициализируется стандартным образом через `src/logger.py`; длительные операции логируют этапы выполнения через `logger.info()`/`logger.error()`

**Длительные операции:**
- Операции, способные заблокировать интерфейс (сетевые запросы, запросы к БД, вызовы `run/`-скриптов, работа с Ollama), выполняются в фоновом потоке (`threading` или аналог) - не в главном потоке обработки событий окна
- Прогресс длительной операции отображается в окне; промежуточные строки прогресса в лог не пишутся (по аналогии с разделом 04.03.01)
- По завершении - итоговый результат через `logger.info()`

**Корректное завершение:**
- Закрытие главного окна корректно освобождает ресурсы (раздел 05.03): закрытие подключений к БД, откат незавершенных транзакций, удаление временных файлов
- Обработка закрытия окна привязывается к сигналу завершения окна (например, `WM_DELETE_WINDOW` в tkinter) - не оставлять orphan-процессов и незакрытых ресурсов

**Кросс-платформенность:**
- Выбранная библиотека GUI обязана работать на Windows и Linux (раздел 03.08)
- Пути к ресурсам интерфейса (иконки, шаблоны) строить через `pathlib.Path` относительно `PROJECT_ROOT` - без захардкоженных разделителей

# 05. Безопасность и надежность

Безопасность, обработка ошибок, корректное завершение, привилегии

---

## 05.01. Безопасность

*Обязательно.*

- Не помещать пароли, токены, ключи API в код и в config.ini
- Использовать файл .env для секретов; загружать через переменные окружения (python-dotenv или аналог)
- Все SQL-запросы выполнять через параметризацию; не конкатенировать строки в текст запроса
- Не включать в лог пароли, токены, ключи API и любые другие секреты - даже в контексте ошибки

---

## 05.02. Обработка ошибок

*Обязательно.*

- При логировании ошибки указывать контекст: имя скрипта, этап выполнения, релевантные входные данные (исключая секреты)
- При критической ошибке завершать выполнение с ненулевым exit code (sys.exit(1) или аналог)

---

## 05.03. Корректное завершение

*Обязательно.*

При завершении скрипта (как штатном, так и по сигналу прерывания) корректно освободить все ресурсы. Пользователь не должен терять данные из-за Ctrl+C.

**Ресурсы для освобождения:**
- Подключения к БД - закрыть соединения (connection.close())
- Открытые файлы - закрыть через контекстные менеджеры (with open(...))
- Временные файлы - удалить из папки debug/ или tmp/
- Незавершенные транзакции - откатить (rollback) при прерывании, не оставлять partial state

**Обработка сигналов:**
- SIGINT (Ctrl+C): перехватить через signal или try/except KeyboardInterrupt; выполнить очистку; вывести сообщение о завершении; выйти с exit(1)
- Не подавлять KeyboardInterrupt молча - пользователь должен видеть что скрипт остановлен

**Паттерн через signal:**
```python
import signal
import sys

def _signal_handler(sig, frame):
    logger.info(t("msg.interrupted"))
    # cleanup
    sys.exit(1)

signal.signal(signal.SIGINT, _signal_handler)
```

Или через try/finally:
```python
try:
    main()
except KeyboardInterrupt:
    logger.info(t("msg.interrupted"))
    sys.exit(1)
finally:
    # cleanup: закрыть подключения, удалить временные файлы
```

**Правила:**
- Не использовать atexit для критичных ресурсов - он не вызывается при sys.exit() из signal handler на некоторых платформах
- Использовать контекстные менеджеры (with) для файлов и подключений - автоматическое закрытие
- Не оставлять orphan-процессов при работе с subprocess

---

## 05.04. Привилегированные операции (root, sudo)

*Обязательно.*

- Если для выполнения задачи агенту требуются права root или sudo (установка системных пакетов, изменение системных конфигов, работа с системными службами и т.п.) - ни в коем случае не выполнять такие команды самостоятельно
- Вместо этого агент должен: сообщить пользователю, какая именно команда требует повышенных привилегий и зачем она нужна; попросить пользователя выполнить ее вручную; дождаться от пользователя подтверждения выполнения; продолжить работу только после подтверждения
- Не использовать `sudo` в командах, запускаемых агентом; не пытаться обойти ограничения через `su`, `pkexec` и аналогичные механизмы

# 06. Документация и версионирование

Git, README, AGENTS, отладка, тестирование

## 06.01. Git

*Обязательно.*

**Инициализация:**
- При начале работы над проектом сразу выполнить `git init` и создать `.gitignore`
- Первый коммит выполнять после того, как код написан и структура проекта сформирована - не в самом начале
- Использовать одну ветку: `master`

**Файл .gitignore** должен исключать: `.env`, `log/`, `debug/`, `diag/`, `input/`, `output/`, `data/`, `__pycache__/`, `.pytest_cache/`, `.venv/`, `.vscode/`, `.claude/`, `.kilo`, а также транзитные подпапки `doc/`: `doc/agent_questions/` и `doc/change_requests/`. Сами папки `doc/standards`, `doc/specs`, `doc/skills` коммитить - они содержат проектные артефакты.

**Сохранение пустых папок в Git:** Git не хранит пустые директории. Для папок, которые исключены в `.gitignore`, но реально используются в проекте (упоминаются в коде, конфиге или структуре проекта), размещать в папке файл `.gitkeep`. В `.gitignore` исключать содержимое папки, но не сам `.gitkeep`:

```
log/*
!log/.gitkeep
```

Так папка сохраняется в репозитории как пустая, а ее реальное содержимое (логи, отладочные файлы и т.д.) не попадает в коммиты. Паттерн применяется ко всем исключаемым папкам с локальным содержимым: `log`, `debug`, `diag`, `input`, `output`, `data`, `doc/agent_questions`, `doc/change_requests`:

```
diag/*
!diag/.gitkeep
doc/agent_questions/*
!doc/agent_questions/.gitkeep
doc/change_requests/*
!doc/change_requests/.gitkeep
```

Файл `.gitkeep` создавать только для тех папок из `.gitignore`, которые реально нужны проекту (log, debug, diag, input, output и др.) - не добавлять `.gitkeep` в папки, которые заведомо не используются.

**Формат сообщений коммитов** - Conventional Commits, язык русский:

```
<тип>(<scope>): <краткое описание>
<тип>: <краткое описание>
```

Scope указывает область изменений (например: `db`, `api`, `config`, `logger`, `docs`). Формат со scope и без scope равнозначны - выбор зависит от контекста изменения. Если изменение изолировано в конкретной области - использовать scope. Если изменение затрагивает несколько областей или не относится к конкретной - коммит без scope.

Типы:

| Тип      | Назначение                                                   |
| -------- | ------------------------------------------------------------ |
| feat     | Новая функциональность                                       |
| fix      | Исправление ошибки                                           |
| refactor | Рефакторинг без изменения поведения                          |
| chore    | Обслуживание: зависимости, конфиги, инструменты              |
| docs     | Изменения только в документации                              |
| test     | Добавление или исправление тестов                            |
| style    | Форматирование, пробелы, переносы - без изменения логики     |

Примеры:
```
feat(api): добавить загрузку данных из API
fix: исправить обработку пустого ответа сервера
chore(deps): добавить зависимости в requirements.txt
docs: обновить README - инструкция по запуску
```

**Правила:**
- Описание - на русском, в повелительном наклонении, с маленькой буквы, без точки в конце
- Одна логическая единица изменений - один коммит; не смешивать несвязанные правки
- Не коммитить файлы из `.gitignore`; не коммитить секреты
- Не упоминать в сообщениях коммитов и комментариях к коду, что изменения сделаны ИИ-агентом, нейросетью или инструментом автоматической генерации кода

**Авторство коммитов AI-агента:** при выполнении коммита агент определяет данные автора динамически из системы и сохраняет в локальный конфиг репозитория:
- `user.name` = вывод команды `whoami`
- `user.email` = `$(whoami)@$(hostname)`
- Не задавать фиксированные значения; всегда вычислять на текущей машине
- При первом коммите: вычислить значения и сохранить в локальный конфиг:
  ```bash
  git config --local user.name "$(whoami)"
  git config --local user.email "$(whoami)@$(hostname)"
  ```
- При последующих коммитах: проверить сохраненные значения в локальном конфиге; если текущие whoami/hostname не совпадают - автоматически обновить локальный конфиг на новые значения
- Не использовать флаги `-c` в команде `git commit`; полагаться на локальный конфиг репозитория

**Состав коммита по запросу пользователя:**
Если пользователь явно просит сделать коммит, включать в него все измененные и неотслеживаемые файлы, которые не входят в `.gitignore`, - независимо от того, кто их создал (агент, скрипт, пользователь). Не ограничиваться только файлами, измененными в рамках текущей задачи агента. Перед коммитом выполнить `git status`, убедиться что ни один файл из `.gitignore` не попадает в индекс, и только затем выполнять `git add` и коммит.

## 06.02. README.md и AGENTS.md

*Обязательно.*

Каждый проект должен содержать `README.md` в корне. Предпочтительный язык - русский. Это инструкции для пользователя: писать на бизнес-языке, без технического жаргона, без упоминания AI-агентов и стандартов. Минимальная структура:

- **Версия стандарта** - строка вида `Соответствует стандарту: X.Y.Z` (семантическая версия `doc/standards/project_standards.md`, по которой создан проект)
- **Назначение** - что делает проект (1-3 предложения)
- **Требования** - версия Python, список зависимостей
- **Установка** - создание окружения, установка зависимостей. Обязательно включать bash-команды для настройки venv:

```bash
python3 -m venv --copies .venv
source .venv/bin/activate
pip install -r requirements.txt
```
- **Настройка** - что заполнить в `.env` (перечень ключей) и `config/config.ini` (перечень параметров)
- **Пайплайн проекта** - описание последовательности этапов обработки данных: какие шаги выполняются, в каком порядке, что поступает на вход и что получается на выходе каждого этапа
- **Структура файлов и директорий** - таблица с описанием назначения каждой папки и каждого файла проекта (все папки и все файлы, без исключения). Формат: первый столбец - путь к папке или файлу, второй столбец - назначение. При актуализации документации по запросу пользователя агент перечитывает весь проект и обновляет информацию о папках и файлах в README.md в соответствии с текущим состоянием проекта
- **Запуск** - какие скрипты запускать и в каком порядке
- **Дерево меню** (опционально - только для проектов с интерактивными меню) - схема навигации в виде древовидной диаграммы. Диаграмма строится по структуре меню из `src/menu.py` и автоматически обновляется при изменении меню. Формат универсальный - использовать плейсхолдеры `{t("key.name")}` вместо конкретных названий, чтобы диаграмма не устаревала при смене переводов. Пример:

  ```
  {t("app.name")}
  ├── [1] {t("menu.section1")}
  │   ├── [1] {t("menu.section1.action1")}
  │   ├── [2] {t("menu.section1.action2")}
  │   └── [0] {t("menu.main_menu")}
  ├── [2] {t("menu.section2")}
  │   ├── [1] {t("menu.section2.action1")}
  │   └── [0] {t("menu.main_menu")}
  ├── [3] {t("menu.section3")}
  │   ├── [1] {t("menu.section3.action1")}
  │   ├── [2] {t("menu.section3.action2")}
  │   │   ├── [1] {t("menu.section3.action2.sub1")}
  │   │   ├── [9] {t("menu.back")}
  │   │   └── [0] {t("menu.main_menu")}
  │   └── [0] {t("menu.main_menu")}
  └── [0] {t("menu.exit")}
  ```

  Корень дерева - название приложения из `config.ini [app] name`. Пункты меню указываются через `{t(...)}` - так же, как в коде `src/menu.py`. Номер `0` в главном меню - выход из скрипта: `{t("menu.exit")}`. В подменю второго уровня (первое подменю) - `0`: `{t("menu.main_menu")}`, номер 9 доступен для функциональных разделов. В подменю третьего уровня и глубже номер `0` - возврат в главное меню: `{t("menu.main_menu")}`, номер `9` - возврат на шаг назад: `{t("menu.back")}`. При изменении структуры меню - диаграмма в README обновляется синхронно.

`AGENTS.md` проекта - основной файл инструкций для AI-агента. Предпочтительный язык - английский. Должен содержать строку `Соответствует стандарту: X.Y.Z` - чтобы агент при последующей работе с проектом знал, по какой версии стандарта он создан.

`AGENTS.md` и `CLAUDE.md` - зеркальные копии (за исключением главного заголовка H1 и описания назначения в начале файла, которые должны соответствовать целевому инструменту: `AGENTS.md` для Kilo Code и Cursor, `CLAUDE.md` для Claude Code). При любом изменении одного из них - синхронно вносить те же изменения во второй. Зеркалированию подлежит только содержательное наполнение (правила, структура, соглашения). Содержание обоих файлов всегда идентично. Основным файлом считается `AGENTS.md`; `CLAUDE.md` - вспомогательный, обеспечивающий совместимость с Claude Code.

**Примеры заголовков и описания назначения:**

`AGENTS.md`:
```
# Инструкции для AI-агентов

Этот файл содержит инструкции для AI-агентов при работе с данным проектом.

Соответствует стандарту: X.Y.Z
```

`CLAUDE.md`:
```
# Инструкции для Claude Code

Этот файл содержит инструкции для AI-агента Claude Code при работе с данным проектом.

Соответствует стандарту: X.Y.Z
```

При редактировании `README.md`, `AGENTS.md` или `CLAUDE.md` (добавление или изменение текста): определить исходный язык документа и писать новый контент на том же языке. Если документ написан по-английски - дополнения на английском; если по-русски - на русском. Не смешивать языки в одном документе.

## 07.01. Общие правила работы

*Обязательно.*

1. Не менять существующую бизнес-логику и поведение без явного согласования с пользователем
2. При необходимости искать решение в интернете
3. После значимых изменений обновлять проектную документацию, включая README.md
4. При обновлении документации: активно выявлять устаревший контент (противоречия, дублирование, неактуальные инструкции). Удалять устаревший контент - только после явного подтверждения пользователя: описать что именно устарело и почему, предложить удалить, дождаться ответа. Без подтверждения - не удалять и не сокращать ничего
5. При запросе пользователя актуализировать документацию: перечитать весь проект (структуру папок, все файлы, конфигурацию), обновить README.md в соответствии с текущим состоянием - в том числе таблицу структуры файлов и директорий, пайплайн, раздел запуска

## 07.02. Общение с пользователем, вопросы и согласование

*Обязательно.*

- Язык общения с пользователем: русский
- Всегда задавать уточняющие вопросы до начала написания кода или внесения изменений; не действовать без предварительного выяснения
- Записывать уточняющие вопросы в файл в папке `doc/agent_questions`; вопросы в чат не задавать
- Имя файла: `questions_[timestamp].md`, где timestamp - момент создания в формате YYYY-MM-DD_HH-MM; каждый новый набор вопросов - новый файл
- Использовать `doc/agent_questions/questions_YYYY-MM-DD_HH-MM.md` как образец структуры файла
- Нумерация вопросов сквозная в рамках файла
- Если для вопроса уместны варианты ответа - перечислять их нумерованным списком непосредственно под вопросом
- Под каждым вопросом оставлять блок для ответа: строка `Ответ:`, затем две пустые строки
- В чат писать только уведомление о том, что создан новый файл с вопросами, с указанием пути к файлу
- В режиме согласования требований: не вносить изменения в проект до получения явного подтверждения пользователя; после записи вопросов - ожидать ответов перед следующим действием
- Задавать вопросы итерационно: после получения ответов анализировать, появились ли новые неясности, и при необходимости задавать следующий набор вопросов; переходить к работе только когда все аспекты задачи полностью понятны

## 07.03. Порядок действий при старте проекта (инициализация)

*Обязательно.*

**Предварительная проверка:** убедиться, что файл `doc/standards/project_standards.md` присутствует в репозитории. Если файл отсутствует - не приступать к работе, а попросить пользователя поместить актуальную версию стандарта в папку `doc/standards/` и дождаться подтверждения.

1. Запросить у пользователя название приложения
2. Получить требования от пользователя, записать в `doc/specs/init_spec.txt`
3. Задать уточняющие вопросы (раздел 07.02); продолжать задавать вопросы итерационно до тех пор, пока не будет полная ясность по всем аспектам задачи - только после этого переходить к следующему шагу
4. Сформировать `doc/specs/final_spec.md`
5. Выполнить `git init`, создать `.gitignore`
6. Создать структуру папок - только те, которые нужны (раздел 01.01)
7. Создать `.env`, `.env.example`, `config/config.ini` (с секцией `[app]`: name от пользователя, version = `0.1.0`), `requirements.txt`, `.python-version`
8. Создать виртуальное окружение `python3 -m venv --copies .venv`, установить зависимости
9. Написать код: `src/`, `run/`, при необходимости `main.py`
10. Проверить работу скриптов
11. Написать `README.md`
12. Сделать первый коммит: `feat: инициализировать проект`

## 07.04. Проверка соответствия стандарту

*Обязательно.*

Агент применяет стандарт на всех этапах работы с проектом: инициализация, разработка, рефакторинг, сопровождение. При любом обращении к проекту агент сверяется с актуальной версией стандарта.

**Если пользователь указал на файл `doc/standards/project_standards.md`** (упомянул в сообщении, сослался на стандарт, попросил проверить) - выполнить полную проверку репозитория на соответствие:

1. Прочитать версию стандарта из заголовка `doc/standards/project_standards.md` (строка вида `Версия X.Y.Z`)
2. Прочитать версию стандарта, по которой создан текущий проект, из `AGENTS.md` или `README.md` (строка вида `Соответствует стандарту: X.Y.Z`)
3. Если версии совпадают - сообщить пользователю, что проект соответствует актуальной версии стандарта, и завершить проверку
4. Если версии отличаются:
   а. Полностью изучить репозиторий проекта: структуру папок, все файлы кода и конфигурации, README.md, AGENTS.md
   б. Полностью изучить актуальный стандарт
   в. Сравнить требования стандарта с текущим состоянием проекта по каждому применимому разделу
   г. Составить список расхождений: что именно не соответствует и в каком разделе стандарта описано требование
   д. Дать рекомендации по рефакторингу: для каждого расхождения - конкретное действие (что изменить, добавить или удалить), приоритет (обязательно / опционально) и ссылку на раздел стандарта
   е. Не вносить никаких изменений без явного подтверждения пользователя

## 07.05. Редактирование файла стандарта

*Обязательно.*

Файл `doc/standards/project_standards.md` является эталонным документом. Правила редактирования зависят от имени текущего репозитория:

- Если имя репозитория - `project_standards`: агент работает в центральном репозитории стандарта и может редактировать `doc/standards/project_standards.md`. Именно здесь вносятся все изменения в стандарт.
- Во всех остальных репозиториях: агент не редактирует `doc/standards/project_standards.md`. Файл доступен только для чтения - агент читает его, чтобы понять, по какой версии стандарта создан проект. Все обновления стандарта вносятся только в центральном репозитории `project_standards`.

**Версионирование** - семантическое (SemVer): `MAJOR.MINOR.PATCH` (например `2.5.0`).

Формат заголовка: `# Стандарт проекта для AI-агента. Версия X.Y.Z. YYYY-MM-DD`

При редактировании обновлять версию и дату по правилам:

- **PATCH** (X.Y.Z → X.Y.Z+1): уточнения формулировок, исправление опечаток, форматирование, добавление элементов в существующие списки
- **MINOR** (X.Y.Z → X.Y+1.0): новые разделы, новые правила, новые требования, структурные изменения
- **MAJOR** (X.Y.Z → X+1.0.0): изменение назначения или области действия документа, несовместимые изменения требований
