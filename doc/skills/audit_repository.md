# Навык: Полный аудит репозитория на соответствие стандарту

## Описание

Независимый read-only аудит произвольного репозитория, построенного по стандарту `project_standards.md`. Агент последовательно изучает стандарты, каждый файл кодовой базы и документацию, сравнивает их между собой и с применимыми разделами стандарта, затем формирует подробный отчет о расхождениях. Правки в код не вносятся.

## Когда использовать

- Пользователь просит проверить репозиторий на соответствие стандарту, провести аудит, ревью соответствия
- Пользователь просит найти расхождения между кодом/документацией и стандартом
- Пользователь запрашивает отчет о состоянии проекта перед релизом, рефакторингом или обновлением стандарта
- Пользоватент указывает на навык явно через `@doc/skills/audit_repository.md`

## Предусловия

- Рабочая директория - корень проверяемого проекта (открывается агентом автоматически)
- В проекте присутствует `doc/standards/project_standards.md` - источник истины
- Доступ на чтение ко всем файлам репозитория

## Приоритеты проверки

Аудит проводится строго в порядке убывания приоритета. Расхождение в более приоритетном слое делает бессмысленным соответствие в нижних:

1. **Стандарты** (`doc/standards/*.md`) - проект должен соответствовать актуальной версии стандарта. Версии в `AGENTS.md`, `CLAUDE.md`, `README.md` должны совпадать с версией в заголовке `project_standards.md`.
2. **Кодовая база** - все файлы кода и конфигурации в репозитории должны соответствовать применимым разделам стандарта. Особое внимание - раздел 04 (вывод и интерфейс).
3. **Документация** (`README.md`, `AGENTS.md`, `CLAUDE.md`) - должна соответствовать уже вычищенной под стандарт кодовой базе: реальные файлы, реальные пути, реальные версии, актуальное поведение.

## Инструкция

### Шаг 0. Подготовка

1. Зафиксировать корень проекта как текущую рабочую директорию. Не использовать абсолютные пути конкретной машины - все пути строить относительно корня проекта.
2. Прочитать ВСЕ файлы из `doc/standards/` целиком. Извлечь версию каждого стандарта из заголовка первой строки в формате `Версия X.Y.Z`. Запомнить версию `project_standards.md` как `STANDARD_VERSION` - она основная.
3. Внимательно изучить ВСЕ разделы всех стандартов. Основной упор на `project_standards.md`, особенно:
   - 01.01 - структура проекта, 01.02 - конфигурация, 01.07 - версионность приложения
   - 04.01 - i18n, 04.02 - логирование, 04.03 - форматирование, 04.04 - маркеры, 04.05 - требования к тексту, 04.07 - ошибки и терминология
   - 05.01 - безопасность, 05.02 - обработка ошибок, 05.03 - корректное завершение
   - 06.01 - git, 06.02 - README и AGENTS
4. Применять к проверяемым файлам только те стандарты из папки, которые релевантны репозиторию: deploy (lin/win), docker, skill, software_doc и любые другие, присутствующие в проекте. Релевантность определять по характеру проекта - например, `docker_deploy_standards.md` применяется только при наличии Docker-инфраструктуры, `skill_standards.md` - при наличии навыков, deploy-стандарт для целевой ОС - при наличии системных инструментов. Нерелевантные стандарты в отчете не проверять. Сверить версии каждого релевантного стандарта с тем, что заявлено в `AGENTS.md`/`CLAUDE.md`/`README.md`.

### Шаг 1. Составить полный список файлов

Собрать все файлы репозитория, не исключенные `.gitignore`, включая точечные файлы (`.env.example`, `.gitignore`, `.python-version`, `pyproject.toml`). Проверять каждый файл на соответствие применимым разделам стандарта. Не пропускать файлы - цель полный охват.

### Шаг 2. Проверка слоя стандартов

- Совпадает ли `STANDARD_VERSION` с записью «Соответствует стандарту: X.Y.Z» (или эквивалентной) в `AGENTS.md`, `CLAUDE.md`, `README.md`
- Синхронны ли версии сопутствующих стандартов между собой и с реестром в `AGENTS.md`/`CLAUDE.md`
- Если проект хранит копию стандартов - актуальна ли она (по версиям в заголовках)

### Шаг 3. Проверка кодовой базы

Особое внимание - вывод и текст (раздел 04):

- **i18n (04.01):** все ли строки вывода проходят через `t()` из `src/localization.py`. Проверить КАЖДЫЙ вызов `logger.info/error/warning/debug`, `print`, `raise` во всех `.py` на наличие строковых литералов на естественном языке.
- **Файл переводов** (`config/translations.json` или таблица БД): все ли ключи, используемые в коде через `t()`, присутствуют в источнике; есть ли мертвые ключи (объявлены, но не вызываются); совпадают ли наборы ключей между языками (ru/en/и т.д.).
- **`t()` в `src/localization.py`:** корректна ли реализация - интерполяция, плюрализация через `count`, fallback-цепочка current -> default -> `!KEY!`.
- **Текст (04.05):** нет ли букв ё/Ё, длинного `—` или короткого `–` тире, многоточий `...` или `…`, лидирующих пробелов в начале строк вывода, точек в конце сообщений лога/консоли, эмодзи, ALL-CAPS акцентов, восклицательных знаков. Проверить и код, и docstring, и комментарии.
- **Логирование (04.02):** каждый скрипт `run/` и `main.py` имеет свой лог-файл; стартовый вывод `app_started` через `t()`; нет `print()` вне разрешенных модулей меню (04.06).
- **Маркеры (04.04):** используются маркеры `[!]`, `[i]`, `[*]` и др. из стандарта, а не самодельные.
- **Форматирование (04.03):** таблицы через `tabulate` с caption; списки с префиксом `- `; сводные блоки; нет строк-разделителей из символов.
- **Ошибки (04.07, 05.02):** контекст в логе (имя скрипта, стадия, релевантные входные данные); критичный сбой завершается `sys.exit(1)`.
- **Завершение (05.03):** обработка SIGINT/KeyboardInterrupt, освобождение ресурсов, context managers, откат транзакций.

Другие проверки кодовой базы:

- **Структура (01.01):** соответствие набора папок; секция `[app]` с `name`, `version`, `language`.
- **Версия приложения (01.07):** стартовый вывод `APP_NAME v{APP_VERSION}` через `t()`.
- **Валидация (01.04):** проверка окружения перед основной логикой; fail-fast с выводом всех проблем сразу.
- **Пути (01.03, 03.08):** `PROJECT_ROOT` от `__file__`/`config.ini`; `pathlib.Path` везде; нет строковой конкатенации путей.
- **Конфиги:** `config.ini` и `config.ini.example` синхронны по набору секций и ключей; `.env.example` синхронен с `.env` по набору переменных.
- **Безопасность (05.01):** секреты только в `.env`, не в `config.ini` и не в коде; секреты не логируются.
- **SQL (03.01):** запросы и DDL в `.sql` под `sql/`; параметризованные запросы; нет конкатенации пользовательского ввода.
- **Идемпотентность (03.05):** безопасность повторного запуска; UPSERT/проверки существования; атомарная запись файлов через `.tmp` + `replace()`.
- **Retry (02.04):** экспоненциальный backoff; нет ретраев auth (401/403) и validation (400/422).
- **Типизация (03.07):** аннотации на всех функциях; `typing`/`from __future__ import annotations` для сложных типов.
- **pyproject.toml:** версия синхронна с `config.ini` (`[app] version`).
- **Run-скрипты (03.02-03.09):** обязательные скрипты присутствуют для задействованных доменов (db/api/ai/email).

### Шаг 4. Проверка слоя документации

- **README (06.02):** перечислены ли все актуальные файлы проекта (`src/localization.py`, `config/translations.json` и т.д.); нет ли упоминаний несуществующих файлов; структура файлов соответствует реальной.
- **AGENTS.md / CLAUDE.md:** синхронны ли между собой (кроме H1 и блока назначения); реестр навыков актуален; версии стандартов совпадают.
- **Согласованность с кодом:** документация описывает реальное поведение вычищенной кодовой базы - не расходится с фактическими ключами переводов, версиями, путями, командами.

### Шаг 5. Проверка git и инфраструктуры

- **`.gitignore` (06.01):** все исключаемые папки с локальным содержимым имеют паттерн `folder/*` + `!folder/.gitkeep`; файлы `.gitkeep` присутствуют в нужных папках; исключены `.env`, `.venv/`, `log/`, `debug/`, `input/`, `output/`, `data/`, `__pycache__/`, `.vscode/`, `.claude/`, `.kilo`.
- **Тесты (06.04):** папка `tests/` не исключена из git; именование `test_[module].py`; `pytest` в зависимостях.
- **Venv (01.06):** `.venv/` исключена; `.python-version` фиксирует версию.

### Шаг 6. Отчет

Записать отчет в файл `doc/agent_questions/audit_{STANDARD_VERSION}_report.md` (если папка `doc/agent_questions/` отсутствует - создать ее). Имя файла содержит фактическую версию стандарта, а не захардкоженную.

Структура отчета:

```markdown
# Отчет аудита соответствия стандарту {STANDARD_VERSION}

Дата: {YYYY-MM-DD}
Проект: {имя проекта из [app] name или имя папки}

## Сводка
- Проверено файлов: N
- Найдено проблем: N (критичных: N, средних: N, мелких: N)

## Детальный список проблем

Для каждой проблемы - нумерованный пункт:
- Файл:путь:строка
- Раздел стандарта: 0X.0Y
- Слой: стандарты / кодовая база / документация / инфраструктура
- Описание проблемы
- Критичность: критично / средне / мелочь
- Рекомендация: конкретное действие

## Проверки без замечаний

Список разделов/файлов, которые соответствуют стандарту.

## Решения

Четкий список: что исправить обязательно, что опционально, что оставить как есть.
```

## Критерии завершения

- Создан файл `doc/agent_questions/audit_{STANDARD_VERSION}_report.md` с заполненной структурой
- В отчете указано количество проверенных файлов и counts по критичности
- Каждая проблема привязана к файлу:строке и разделу стандарта
- В чат пользователю выдана краткая сводка находок и путь к файлу отчета
- В код репозитория не внесено ни одной правки

## Принципы аудита

- **Дотошность:** цель - найти реальные расхождения, а не подтверждать соответствие.
- **Сомнение = вопрос:** если неоднозначно трактуется - отметить как вопрос, не как нарушение и не как соответствие.
- **Read-only:** никаких правок в коде, конфигах, документации. Только чтение, анализ, отчет.
- **Приоритеты:** нарушение в слое стандартов важнее нарушения в документации; указывать слой для каждой проблемы.
- **Конкретика:** рекомендация должна быть действием, а не констатацией (не «улучшить», а «заменить литерал на `t(\"msg.foo\")` и добавить ключ в `config/translations.json`»).

## Примеры

1. Пользователь: «проверь репо на соответствие стандарту» -> агент читает стандарт, проходит все файлы, пишет `audit_X.Y.Z_report.md`, выдает сводку в чат.
2. Пользователь: «проведи аудит перед релизом» -> агент фокусируется на критичных расхождениях (безопасность, секреты, версии), помечает блокирующие релиз проблемы отдельным списком в разделе «Решения».
3. Пользователь тегает навык (`@doc/skills/audit_repository.md`) -> агент запускает субагента с этой инструкцией, дожидается завершения и файла отчета, после чего выдает пользователю краткую сводку находок и путь к отчету. Сам агент аудит не выполняет - вся работа делегируется субагенту.

## Ограничения

- Не вносить правки в проверяемые файлы - это read-only аудит
- Не запускать скрипты проекта, не выполнять миграции, не трогать БД
- Не редактировать `doc/standards/project_standards.md` - он источник истины, даже если в нем есть недочеты (о них - в отчет, не в код)
- Если проект не содержит `doc/standards/project_standards.md` - сообщить пользователю, что навык применим только к проектам на базе данного стандарта, и остановиться
