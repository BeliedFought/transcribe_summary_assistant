# Transcribe & Summary Assistant

Соответствует стандарту: 3.9.0

## Назначение

CLI-инструмент для транскрибации (speech-to-text), диаризации (разделение по голосам), саммаризации аудио- и видеофайлов и опциональной рассылки саммари на email (SMTP). Пользователь помещает файлы в `input/audio/`, `input/video/` или `input/yt/`, запускает `main.py` - инструмент транскрибирует речь через faster-whisper (CUDA), разделяет по дикторам через pyannote.audio, формирует саммари через DeepSeek API и при необходимости отправляет его на почту. Каждая обработка - отдельная сессия с метаданными в SQLite.

## Требования

- Python 3.14
- NVIDIA GPU с CUDA (минимум 10 GB VRAM для модели large-v3)
- ffmpeg
- yt-dlp

Зависимости перечислены в `requirements.txt`.

## Установка

```bash
python3 -m venv --copies .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Настройка

### .env (секреты)

| Переменная | Назначение |
|-----------|-----------|
| `DEEPSEEK_API_KEY` | API-ключ DeepSeek для саммаризации |
| `HF_TOKEN` | Hugging Face токен для pyannote.audio |
| `SMTP_USER` | Логин SMTP (полный email) для отправки саммари на почту |
| `SMTP_PASSWORD` | Пароль приложения SMTP для отправки саммари на почту |

### config/config.ini (настройки)

Основные секции:
- `[app]` - название, версия, язык (ru/en)
- `[processing]` - поддерживаемые расширения файлов
- `[whisper]` - настройки faster-whisper (модель, устройство, VAD)
- `[diarization]` - настройки pyannote.audio (модель, число дикторов)
- `[youtube]` - настройки yt-dlp для загрузки YouTube
- `[deepseek]` - настройки DeepSeek API
- `[ollama]` - настройки Ollama (отключено по умолчанию)
- `[email]` - рассылка саммари и статьи на почту (SMTP, получатель, темы `subject_summary`/`subject_article`; `enabled=false` по умолчанию)
- `[output]` - какие документы формировать: `generate_summary`, `generate_article`
- `[paths]` - пути к папкам проекта
- `[retry]` - настройки повторных попыток

## Пайплайн проекта

1. Сканирование папок `input/audio/`, `input/video/`, `input/yt/`
2. Из `.txt` файлов `input/yt/` извлекаются все ссылки YouTube - на каждую ссылку создается отдельная сессия (повторяющиеся ссылки дедуплицируются)
3. Загрузка аудио для каждой YouTube-ссылки через yt-dlp
4. Создание сессии (уникальный ID, папка, запись в SQLite)
5. Для видео: извлечение аудио через ffmpeg (16kHz, mono WAV)
6. Транскрибация через faster-whisper large-v3 (CUDA, float16, VAD)
7. Диаризация через pyannote.audio (определение дикторов)
8. Алайнмент - привязка сегментов транскрипции к дикторам
9. Сохранение транскрипции с ролями в папку сессии
10. Саммаризация через DeepSeek API (общий контекст + по ролям)
11. Генерация статьи через DeepSeek API (краткое изложение для публикации)
12. Сохранение саммари и статьи + копии в `output/`
13. Отправка саммари и статьи на email (если включено в `[email]`): каждое - отдельным письмом

Этапы 3, 6 и 7 кешируются (см. раздел [Кеширование](#кеширование)): повторная обработка того же файла не использует GPU и не повторяет загрузку аудио.

Источники обрабатываются последовательно: сначала все аудио, затем видео, затем YouTube-ссылки. В начале запуска выводится сводка источников к обработке (количество по типам).

## Кеширование

Чтобы повторная обработка того же файла не нагружала GPU и не повторяла сетевые запросы, кешируются три ресурсоемких этапа. Кеш привязан к хешу содержимого аудиофайла (`file_hash` в БД): совпадение хеша означает тот же аудиоматериал.

| Этап | Что кешируется | Файл сессии | Когда пропускается |
|------|---------------|-------------|--------------------|
| Загрузка YouTube-аудио | Скачанный аудиофайл | `debug/yt_audio/` | Найдена прежняя сессия с тем же video ID и целым аудиофайлом |
| Транскрибация (whisper, GPU) | Сегменты + язык + уверенность | `transcription.json` | Найдена сессия с тем же `file_hash` и валидным `transcription.json` |
| Диаризация (pyannote, GPU) | Сегменты дикторов | `diarization.json` | Найдена сессия с тем же `file_hash` и валидным `diarization.json` |

Таким образом, перегенерация саммари и статьи по ранее обработанному материалу проходит полностью без GPU и без загрузки аудио - выполняются только запросы к DeepSeek API. Поиск кандидатов в кеше идет от свежей сессии к старой; берется первый, у которого на месте нужный JSON-файл. Первый прогон нового файла всегда прогоняет whisper и диаризацию на GPU и заполняет кеш для последующих запусков.

Вид формируемых документов настраивается в секции `[output]`: `generate_summary` и `generate_article` (оба включены по умолчанию).

## Формат саммари

Шапка файла `summary.md`:

- **YouTube**: канал, количество подписчиков, ссылка на видео и канал
- **Аудио/видео**: имя файла, длительность, размер

При недоступности количества подписчиков выводится `н/д`.

## Формат статьи

Статья (`article.md`) - краткое структурированное изложение на русском языке, готовое к публикации в Telegram или на сайте. Содержит заголовок первого уровня (`#`), который формирует модель по содержанию, и подзаголовки (`##`). В отличие от саммари, статья не содержит метаданных, технических пометок, разделения по дикторам и указания источников - это чистый документ для публикации.

## Рассылка на email

После каждой успешной обработки саммари и статья отправляются получателю из `config.ini` (`[email] to`). Каждое отправляется отдельным письмом с собственной темой: тема саммари - `[email] subject_summary`, тема статьи - `[email] subject_article`. Отправка включается флагом `[email] enabled`. Учетные данные SMTP берутся из `.env` (`SMTP_USER`, `SMTP_PASSWORD`). Письмо формируется в формате multipart: HTML-часть с оформленным документом (конвертация markdown в HTML с таблицами, блоками кода и цитатами) и plain text-часть с исходным markdown как fallback. Статусы отправки сохраняются раздельно: `email_status` (саммари) и `article_email_status` (статья). При ошибке отправки логируется ошибка, обработка продолжается.
